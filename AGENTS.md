# AGENTS.md

このリポジトリでは、研修用の「ミニ CSV 集計サービス」を段階的に実装します。

## 回答方針

- 日本語で簡潔かつ丁寧に回答する
- 作業内容、確認結果、未実施事項を明確に伝える
- 研修用の読みやすさを優先し、過度に複雑な実装を避ける

## ブランチ運用

- 各 step の完成状態をブランチとして残す
- ブランチ名は以下を基本とする
  - `step0-docs`
  - `step1-backend-base`
  - `step2-db-base`
  - `step3-frontend-base`
  - `step4-delete`
  - `step5-register`
  - `step6-detail`
  - `step7-aggregate`
- 既存 step の完成状態を壊す変更は避ける
- 次 step を始めるときは、前 step の完成ブランチを元に新しいブランチを作る

## 実装方針

- Docker Compose でバックエンド、フロントエンド、DB を起動できる構成にする
- バックエンドは Python / Django を使う
- フロントエンドは TypeScript / Angular を使う
- DB は PostgreSQL を使う
- 各 step では、その step の目的に必要な変更だけを行う

## データ設計方針

- CSV のメタデータは `items` テーブルで管理する
- CSV の実データは PostgreSQL 上の専用スキーマに個別テーブルとして保存する
- `items.table_name` には実データテーブル名を保存する
- `table_name` はユーザー入力ではなく、衝突しない安全な値を生成する

## CSV バリデーション方針

- 1 行目をヘッダーとして扱う
- 列名は英数字とアンダースコアのみ許可する
- 列数は最大 20 とする
- 行数は最大 1000 とする
- 空 CSV は登録不可とする
- 重複列名は登録不可とする
- 列の型は文字列と整数のみ扱う

## セキュリティと DB 操作

- SQL インジェクション対策を重視する
- ユーザー入力を SQL 文字列に直接埋め込まない
- テーブル名、列名、集計関数は許可リストまたは安全に生成した識別子のみ使う
- メタデータ登録、実データテーブル作成、データ投入はトランザクションで扱う
- 削除時はメタデータと実データの不整合が残らないようにする

## テスト方針

- 各 step で最低限の疎通確認を行う
- バックエンド API は正常系と主要な異常系を確認する
- CSV 登録ではバリデーション失敗ケースを確認する
- フロントエンドでは主要画面の表示と API 呼び出しを確認する
